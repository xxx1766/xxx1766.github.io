OpenClaw 是一个开源、自托管的 AI 代理框架，它将大型语言模型（LLM）转变为持久化、能够使用工具并与现实世界集成的助手。与仅仅作为 API 调用代理的聊天机器人不同，OpenClaw 实现了一个完整的代理运行时，包括会话管理、记忆持久化、上下文窗口优化、多渠道消息传递、沙盒工具执行以及事件驱动的扩展性。

本文深入探讨了 OpenClaw 的工作原理——不仅仅是它所运行的 AWS 基础设施，更重要的是那些让无状态 LLM 表现得像一个有状态、持续可用的助手的软件架构决策。

高层架构

以下是 OpenClaw 的高层架构概览：

网关——中央神经系统

网关是一个单独的、长期运行的 Node.js 守护进程，负责管理所有状态和连接。可以将其视为应用服务器——所有数据流都经过它。

WebSocket 优先协议

网关在可配置端口（默认 127.0.0.1:18789）上暴露了一个类型化的 WebSocket API。所有客户端——macOS 应用、CLI、Web UI、移动节点和自动化——都通过这个单一的 WebSocket 进行连接。

有线协议：

  • 传输：WebSocket，JSON payload 的文本帧
  • 第一个帧必须是 connect 握手
  • 请求：{type:"req", id, method, params} → {type:"res", id, ok, payload|error}
  • 事件：{type:"event", event, payload, seq?, stateVersion?}
  • 对于有副作用的方法（send, agent），需要幂等键以实现安全重试

连接生命周期：

设备配对与信任模型

所有 WebSocket 客户端在连接时都会声明一个设备身份。网关实现了信任模型：

  • 本地连接（环回/同一主机的 Tailnet）可以自动批准
  • 非本地连接必须签署一个挑战随机数并需要明确批准
  • 配对后会颁发设备令牌，用于后续重连
  • 网关认证令牌（OPENCLAW_GATEWAY_TOKEN）适用于所有连接

渠道桥接

网关维护与消息平台的持久连接：

  • WhatsApp 通过 Baileys（web 协议）
  • Telegram 通过 grammY
  • Discord、Slack、Signal、iMessage 通过各自的 SDK
  • WebChat 静态 UI 使用网关 WS API

每个桥接器将平台特定的事件转换为标准化的内部信封。关键的架构洞察是：网关是唯一持有消息会话的进程——每个主机只有一个 WhatsApp 会话，一个 Telegram 机器人连接等。

代理循环——消息如何变为行动

当收到入站消息时，完整的生命周期如下：

接收与路由

Inbound message → Channel Bridge → Session Resolution → Command Queue → Agent Runtime

入站消息 → 渠道桥接 → 会话解析 → 命令队列 → 代理运行时

  1. 渠道桥接器标准化消息（发送者、内容、附件、线程上下文）
  2. 会话管理器根据 dmScope 规则解析会话键：
    • main：所有私信共享一个会话（跨设备/渠道的连续性）
    • per-peer：按发送者 ID 隔离
    • per-channel-peer：按渠道 + 发送者隔离（推荐用于多用户）
  3. 命令队列序列化运行

命令队列

这是一个支持车道的 FIFO 队列，用于防止并发代理运行冲突：

队列模式控制入站消息如何与活动运行交互：

  • **collect**（默认）：将排队的消息合并为一个后续回合
  • **steer**：注入当前运行，在下一个边界取消待处理的工具调用
  • **followup**：等待当前运行结束，然后开始新的回合
  • **steer-backlog**：立即转向，并为后续保存

提示词组装

OpenClaw 为每个代理运行构建一个自定义系统提示词。这并非静态字符串——它由多个来源组装而成：

关键设计决策：

  • 引导文件会在 bootstrapMaxChars（默认 20,000 字符）处截断，以保持提示词精简
  • 技能仅作为元数据列出（名称 + 描述 + 文件路径）——模型会根据需要 read 完整的 SKILL.md
  • 工具 Schema（JSON）随附发送，但即使它们不是可见文本，也计入上下文
  • 时间仅包含时区（无动态时钟），以保持提示词在回合之间缓存稳定

模型推理与流式传输

嵌入式运行时（pi-mono）处理实际的 LLM 交互：

  1. 解析模型 + 认证配置文件
  2. 通过每个会话 + 全局队列序列化运行
  3. 将助手增量流式传输为 event:agent 帧
  4. 工具调用在推理回合之间执行（代理循环）
  5. 强制执行超时（默认 600 秒）

流式传输架构：

  • 助手文本增量实时流式传输
  • 工具启动/更新/结束事件在单独的 tool 流上发出
  • 块流式传输可以在完成时立即发出已完成的块（可配置的块大小为 800-1200 字符，优先选择段落中断）
  • NO_REPLY 是从传出有效载荷中过滤掉的哨兵令牌（实现静默回合）

工具执行

工具是代理的“手”。核心工具始终可用（受策略约束）：

  • read / write / edit — 文件操作
  • exec / process — shell 命令执行 + 后台进程管理
  • browser — 浏览器自动化（基于 CDP）
  • web_search / web_fetch — 网络访问
  • message — 跨渠道消息传递
  • cron — 定时任务管理
  • memory_search / memory_get — 语义记忆检索
  • sessions_spawn / sessions_send — 子代理编排
  • nodes — 配对设备控制（摄像头、屏幕、位置、运行）
  • canvas — UI 画布控制
  • tts — 文本转语音
  • gateway — 自我管理（重启、配置、更新）

记忆架构——无状态模型如何记忆

这可能是架构上最有趣的部分。LLM 没有持久记忆——每个会话都从零开始。OpenClaw 通过分层记忆系统解决了这个问题：

记忆栈

作为记忆的工作区文件

记忆是代理工作区中的纯 Markdown 文件。这些文件是真相的来源；模型只“记住”写入磁盘的内容。

  • **memory/YYYY-MM-DD.md**：每日日志文件（仅追加）。会话开始时会读取今天和昨天的日志。
  • **MEMORY.md**：精选的长期记忆。仅在主私有会话中加载（从不在群组上下文中——安全边界）。
  • **AGENTS.md**：操作说明，每个会话都会注入。
  • **SOUL.md**：角色和边界，每个会话都会注入。

基本洞察：代理被指示在学习到值得记住的内容时写入这些文件。“心理笔记”不会在会话重启后存活——文件会。

向量记忆搜索（语义召回）

OpenClaw 为记忆文件构建向量索引以进行语义搜索：

索引管道：

混合搜索 (BM25 + Vector)：

  • 向量相似度：语义匹配（“Mac Studio gateway host”匹配“the machine running the gateway”）
  • BM25 关键词相关性：精确令牌匹配（ID、代码符号、错误字符串）
  • 分数组合：finalScore = vectorWeight × vectorScore + textWeight × textScore
  • 默认权重：70% 向量，30% BM25

嵌入提供商（自动选择并带回退链）：

  1. 通过 node-llama-cpp 的本地 GGUF 模型（约 0.6 GB）
  2. OpenAI text-embedding-3-small
  3. Google Gemini gemini-embedding-001

**重新索引触发器：**索引存储嵌入提供商/模型 + 端点指纹 + 分块参数。如果任何更改，OpenClaw 会自动重置并重新索引。

暴露给代理的工具：

  • memory_search — 语义搜索，返回带有文件 + 行范围的片段
  • memory_get — 按路径读取特定的记忆文件内容

预压缩记忆刷新

当会话接近自动压缩（上下文窗口即将满）时，OpenClaw 会触发一个静默的代理回合，提醒模型在上下文被总结之前将持久笔记写入磁盘：

这由软阈值令牌控制，每个压缩周期最多运行一次。

会话管理——跨对话的连续性

会话键与路由

每个对话都映射到一个会话键：

会话生命周期

  • 每日重置：默认当地时间凌晨 4:00
  • 空闲重置：可选滑动窗口（idleMinutes）
  • 手动重置：/new 或 /reset 开始新会话

持久化

  • 转录本：~/.openclaw/agents/<agentId>/sessions/<SessionId>.jsonl 中的 JSONL 文件
  • 存储：sessions.json——一个 sessionKey → {sessionId, updatedAt, ...} 的映射
  • 来源元数据：每个会话记录其来源（渠道、发送者、线程等）

上下文窗口与压缩

每个模型都有有限的上下文窗口。当它填满时：

压缩与修剪：

  • 压缩：总结并持久化在转录本中（永久）
  • 会话修剪：每次请求时，内存中修剪旧的工具结果（非破坏性）

技能系统——教代理新技能

技能是代理的学习能力——用于特定工具和任务的结构化指令集。

架构

延迟加载模式

技能不会注入到提示词中。只注入一个紧凑的元数据列表：

模型被指示**仅当任务匹配时才 ****read**``**SKILL.md**。这使得提示词开销最小（每个技能约 97 字符 + 字段长度），同时仍能实现有针对性的工具使用。

7.3 门控（加载时过滤器）

技能可以在 YAML frontmatter 中声明要求：

  • **requires.bins**：PATH 上必须存在的二进制文件
  • **requires.env**：必须设置的环境变量
  • **requires.config**：必须为真的配置路径
  • **os**：平台限制（darwin, linux, win32）

技能在会话开始时进行快照，并重用于后续回合（文件系统更改时热重载）。

子代理架构——并行工作

子代理支持后台并行执行，而不会阻塞主对话。

设计

关键设计决策

  • 会话隔离：每个子代理获取 agent:<agentId>:subagent:<uuid>
  • 无嵌套：子代理不能生成子代理（防止扇出）
  • 受限工具：默认情况下没有 sessions_list、sessions_history、sessions_send、sessions_spawn
  • 通知模式：结果会以状态、运行时、令牌使用和成本的形式发布回请求者聊天
  • 自动归档：会话在可配置的超时后归档（默认 60 分钟）
  • 并发性：专用车道，可配置最大值（默认 8）
  • 成本控制：可以通过 agents.defaults.subagents.model 为子代理使用更便宜的模型

心跳与 Cron——主动行为

心跳

心跳是主会话中周期性的代理回合。网关每隔 N 分钟（默认 30 分钟）会根据可配置的提示词发出一个心跳。

响应契约：

  • HEARTBEAT_OK = 无需关注（被抑制，从不传递）
  • 任何其他内容 = 警报内容（传递到配置的渠道）
  • HEARTBEAT.MD 是代理在每次心跳时读取的可选工作区清单

Cron

对于需要精确时间或隔离的任务：

  • 调度类型：一次性（at）、重复（every）、cron 表达式（cron）
  • Payload 类型：systemEvent（注入主会话）或 agentTurn（隔离会话）
  • 传递：将结果通知到特定渠道/收件人
  • 隔离的 cron 作业每次运行都会生成新的会话

何时使用：

心跳 Cron
批量检查 需要精确时间
需要主会话上下文 需要隔离
时间可能漂移 不同的模型/思考
减少 API 调用 一次性提醒

沙盒——减少爆炸半径

OpenClaw 可以在 Docker 容器内运行工具执行，以限制损害。

模式

  • **off**：工具直接在主机上运行
  • **non-main**：仅沙盒非主会话（群聊、子代理）
  • **all**：每个会话都在沙盒中运行

范围

  • **session**：每个会话一个容器
  • **agent**：每个代理一个容器
  • **shared**：所有沙盒会话共享一个容器

工作区访问

  • **none**：工具看到一个沙盒工作区（无主机访问）
  • **ro**：主机工作区以只读方式挂载
  • **rw**：主机工作区以读写方式挂载

逃逸口

  • **tools.elevated**：明确的主机执行绕过（针对授权发送者）
  • 工具策略：在沙盒规则之前仍应用允许/拒绝规则
  • 自定义绑定挂载：暴露特定的主机目录

安全架构

分层安全模型

系统提示词安全

系统提示词包含咨询性防护措施：

  • 无独立目标（无自我保护、复制、资源获取）
  • 优先考虑安全和人工监督，而非完成任务
  • 遵守停止/暂停/审计请求
  • 不操纵以扩大访问权限

这些是咨询性的（模型行为指导）。硬性强制执行使用工具策略、执行批准、沙盒和渠道白名单。

会话隔离

  • dmScope: "per-channel-peer" 推荐用于多用户设置
  • MEMORY.md 仅在私有会话中加载（从不在群组上下文中）
  • 子代理获取受限工具集
  • 每个会话的沙盒防止跨会话文件系统访问

此架构的独到之处（对于架构师而言）

有状态性问题

LLM 是无状态函数：f(prompt) → response。OpenClaw 的核心挑战是通过以下方式使其感觉有状态：

- **基于文件的记忆**，在进程重启后仍然存在
- **会话转录本**（JSONL），重放对话历史记录
- **预压缩记忆刷新**，在上下文丢失之前提取持久知识
- **向量搜索**，实现跨会话的语义召回

上下文窗口作为资源

上下文窗口是代理的“工作记忆”——有限且昂贵。OpenClaw 像缓存一样管理它：

- **压缩** = 带总结的逐出
- **修剪** = 修剪低价值数据（旧工具结果）
- **技能延迟加载** = 指令的按需分页
- **引导截断** = 注入文件的尺寸限制

单写入器架构

网关是每个会话的单写入器系统。命令队列确保：

- 每个会话一次只有一个代理运行
- 无并发写入会话转录本
- 多渠道消息的确定性排序

工作区作为真相来源

代理工作区是一个可由 Git 管理的纯 Markdown 文件目录。这意味着：

- 记忆是人类可读和可编辑的
- 版本控制提供审计跟踪
- 无需管理数据库（SQLite 仅用于向量搜索）
- 迁移是 `git clone` + 配置更新

技能作为按需加载的指令

OpenClaw 不会将所有工具指令嵌入到每个提示词中（令牌开销大），而是将技能作为元数据列出，并让模型按需 read 它们。这类似于开发人员拥有一个带有文档的 IDE：您不会在启动时阅读所有文档，而是在需要时查找所需内容。

总结：构建模块

概念 实现 重要性
持久化 Markdown 文件 + JSONL 转录本 人类可读、可 Git 管理、无数据库
记忆 文件层 + 向量搜索 + 刷新 无状态模型表现为有状态
上下文管理 压缩 + 修剪 + 延迟加载技能 高效利用有限的令牌窗口
并发 基于车道的命令队列 无锁的安全序列化
可扩展性 钩子 + 插件 + 技能 无需分支即可修改行为

多渠道 桥接模式 + 会话路由 一个代理，多个界面
主动行为 心跳 + cron 代理无需被要求即可行动
并行工作 带通知的子代理 后台任务不阻塞聊天
物理世界 节点配对 + 设备命令 代理超越屏幕进行交互
